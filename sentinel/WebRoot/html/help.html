Introduction

Important changes between the Sentinel Tool release 3.1 and 3.2 are indicated by the heavy blue line to the left of the paragraph. All window images and report examples are updated for the 3.2 interface. More information concerning changes can be found in the caCORE 3.2 Release Notes.

The caDSR Sentinel Tool is a web-based tool that allows users to create and manage Alert Definitions for the caDSR database. Alert Definitions are a set of rules that are periodically evaluated against the caDSR. If the conditions in those rules are met, notification is sent to the user (via e-mail) in the form of a hyperlink to an Alert Report. Alert Reports detail the changes (activities) that have taken place within the database.

The caDSR Sentinel Tool provides users with the ability to monitor all changes to Administered Components (Data Elements, Data Element Concepts and Value Domains). The Administered Components of interest are specified through the selection of one or more of the following criteria:

  • Context
  • Protocols
  • Forms / Templates
  • Classification Scheme / Classification Scheme Item
  • Administered Component Type
  • Created By (Name)
  • Modified By (Name)

Users can also specify Administered Component attributes to monitor, including:

  • Workflow Status
  • Registration Status
  • Version

Additional details on the processing and interpretation of an Alert Definition can be found in the Sentinel Tool Installation Guide, System Administrator Notes section. Please direct questions, comments and suggestions to the NCICB Help Desk or use one of the Forums on the NCICB GForge site under the caDSR Sentinel Tool project.

Note: Within the Sentinel Tool pages please do not use the browser "Back" button. Each page provides the necessary links and buttons for navigation.

Accessing and Using the Sentinel Tool

Enter the URL, http://cadsrsentinel.nci.nih.gov, into the Browser address bar or select the link from the CDE Curation Tool, Options menu. The application login page will open.

Note: Only Internet Explorer Version 6.0 or higher has been certified.

Login Page

The caDSR Sentinel Tool requires user authentication. By design, users must have a caDSR account to view, create, edit and run Alert Definitions. At NCICB, the Guest account is not valid for the Sentinel Tool. To obtain a user account for the Sentinel Tool, please contact the Alert or System Administrator for the URL on which it was installed. Alerts List Page

The main page within the Sentinel Tool is the Alerts List page. All features of the tool are accessible from this page. When a task completes, the tool often returns to this page. Initially, this page opens displaying either an empty list with a Welcome message or the Alerts that you have created. From this page, you can also create a new Alert, view all Alerts in the database, and Edit an existing Alert.

The buttons on the Alerts List page perform actions on one or more Alert Definitions in the list. These buttons are enabled and disabled appropriately as items are selected.

Create
Click to create a new Alert Definition. This opens a new page with fields to create a new Alert Definition, see Create for a complete description.
Edit
To enable, select exactly one Alert from the list. The Edit page opens allowing changes to the selected Alert. See Edit for a complete description.

Note: Use of this button is restricted to the Alert creator.
Create Using
To enable, select exactly one Alert. The Edit page opens using the information from the selected Alert with some properties modified to reflect the new state. See Create Using for a complete description.
Delete
To enable, select one or more Alerts. A confirmation message box opens to avoid accidental deletion. Once confirmed, the selected Alerts are permanently deleted from the database.

Note: This button is restricted to the Alert creator.

Note: To exclude an Alert in the Auto Run Process, it is not necessary to delete it. See Edit for details on the eligibility of an Alert for Auto Run.
Run
To enable, select exactly one Alert. The Run Submission page opens allowing the selection of run options prior to submission to the server. See Run for a complete description of the page.

Note: Use of this button is restricted to the Alert creator.
Show All / Show Private
This button switches between "Show All" and "Show Private". Pressing "Show All" displays all alerts defined in the database and changes the button label to "Show Private". Pressing "Show Private" displays alerts that were created by the current user and changes the button label to "Show All".
Help
Appears on every page and displays this file in a separate browser window.
Logout
Appears on every page. Terminates the current Sentinel Tool session and returns to the Login page.

To select or deselect an Alert, click the checkbox to the left of its name. You may also select or deselect all Alerts in the list via the checkbox at the top of the list.

The column headings may be selected to sort the list. Selecting a heading once sorts the list in ascending order; selecting the same heading again sorts in descending order. To perform a more complex sort, select the desired columns in reverse order. For example, to sort the list by Name and Last Auto Run, first select the column heading for the Last Auto Run and then select the column heading for Name. By default the List is initially sorted by Name.

The columns are:

Name
The name is provided by the Creator on the Edit Properties page. Names are not required to be unique. Alert Definitions are created with a hidden unique identifier to allow editing names at the creator's discretion.
Summary
The summary is generated from the content of the Alert Definition when a Save is performed. This column can not be used for sorting.
Frequency
The frequency is selected by the Creator on the Edit Properties page. It defines when the Alert Definition is processed by the Automated Alert Report Generator.
Last Auto Run
The date/time is set by the Sentinel Automated Alert Report Generator when the Alert Definition is processed. This date is set whether or not a report is generated. For example, the frequency of an Alert Definition is set to daily but the date/time shows a date from more than a week ago. Upon inspection of the Alert Definition, note the Start and End dates are set and the Active period ended on the date of the Last Auto Run. A blank date/time and the string "(not yet run)" indicates a report has never been created for this Alert by the Auto Run Process.
Active
The active state of an Alert Definition appears as a check mark when the Edit Properties shows a state other than Inactive. An 'x' will appear when the state is set to Inactive on the Edit Properties page.
Creator
The Alert Definition Creator Login ID.

Edit Page

The Edit page is used to view and maintain the Alert Definition attributes. Select the appropriate action button after completing any changes.

Save
Stores the Alert Definition in the caDSR. Upon successful completion, the List page appears in the browser window.

Restores the definition from the caDSR when viewing an existing Alert Definition.

The Alert Definition attribtues are displayed on four (4) tabs.

Properties
Displays the name, created by, created date, modified date and frequency settings.
Report Details
Displays the report content and recipients.
Criteria
Displays the scope of the data to include in the output.
Monitors
Displays the activities that will trigger a report creation.

Δ Properties Tab

The Properties Tab contains the basic Alert Definition properties.

Alert Name
The Alert Definition identification used both in the notification e-mail and the report body. Use names to briefly describe the function or purpose of the Alert, for example, "Changes to CCR" and "Watch for Retired Records".
Summary
Provides an overview of the Criteria and Monitors. Any reports generated, based on the current report definition, will only include information that meets the description shown.
Auto Run Options
Control the Alert Definition eligibilty in the Automated Report Generation Process by Frequency and Status. Running the Alert manually ignores both Frequency and Status.
Frequency
The execution cycle in the Automated Report Generation Process. The process runs daily at a time determined by the System Administrator. Alerts marked "Daily" are evaluated with each run. Those marked "Weekly" are evaluated only on the specific day of the week selected. Alerts marked "Monthly" are evaluated only on the specific day of the month selected. To indicate the first day of the month, select "1". To indicate the last day of the month, select "31". The Alert may be run manually at anytime via the Run button.
Status
Controls the Auto Run Frequency application in the Automated Report Generation Process.
Active, no end date
The Alert Definition is processed according to the Frequency.
Active until the first report …
After sending a report containing some activity, the status automatically changes to Inactive.
Active only from …
The Alert Definition is processed only within the dates specified. When the begin date is blank, the Alert Definition creation date is used. When the end date is blank and after reaching the begin date, the effect is the same as "Active, no end date".

Note: Midnight is used for all comparisons, e.g. 03/01/2006 to 03/02/2006 is the 24 hour period beginning midnight 03/01/2006.
Inactive
The Alert Definition is retained in the caDSR and not processed by the Automated Report Generation. A brief description is required.

Δ Report Details Tab

The Report Details Tab contains the properties controlling heading content and distribution.

Content
Define the detail appearing on the report.
Include information …
Select to include an expanded header at the top of the report. The expanded header contains all the information appearing on the Properties Tab in addition to the standard information which includes the reporting date and the source database.
Maximum number of levels …
Set the stopping point on reporting the "Associated To" records. Each change in the caDSR has a defined area of effect. The record containing the change is referred to as level 0 (zero). Records directly associated to it are a level 1 (one). Records associated to the associated records are a level 2, and so on. For example, a Value Domain is changed, it is level 0. The Data Elements referencing the Value Domain are a level 1. The Modules, Class Scheme Items, etc that reference the Data Elements are a level 2. And so it continues throughout the database. Contexts appear at many levels due to the direct association between the Context and other records. Continuing the previous example, the Value Domain is directly associated to a Context and so it will be a level 1. The Data Elements associated to the Value Domain are directly associated to Contexts and so it is also a level 2. This can also be visiualized as a series of rings originating at the changed record and expanding out.

Note: All Alert Definitions created prior to the Sentinel Tool 3.2 release have a default of nine (9), show all collateral errects.
Distribution
Define when and to whom reports are sent.
Send a report from …
Choose whether a report is sent with every processing of the Alert Definition or sent only when there is activity. To reduce e-mail traffic, select "… only when there is acitivity …".
Recipients
The e-mail distribution list for Automated and Manually generated reports. There are no limits on the list size or content. Add a recipient e-mail address by selecting a User Name, a Curators group or typing the address. A Process URL may also be entered.
Users
The user accounts defined within the caDSR. The list displays the user name, suffixed first by an asterisk when the account is missing the e-mail address and suffixed by the associated Curator group names as appropriate within parenthesis. If the e-mail address is missing the user may still be added as a Recipient. When the account profile is updated with an e-mail address, the user will begin receiving Alert Reports.
Curators
The Curator groups derived from the user account profiles. A user is included in a Curator group when the account is given any write permissions to the caDSR. Read only accounts are not included in any group. Selecting a Curator group as a recipient hides changes in the organization or assigned responsibilities. Alert Report e-mails to Curator groups are sent to all members except those listed below "Additional e-mail address".
Additional e-mail address and Process URL
A desired recipients well formed e-mail address or software well formed URL. This is often used to specify alternate e-mail addresses, temporary recipients and software process applications. The user account profile supports a single e-mail address. Additional e-mail addresses must be entered here, e.g. for John Smith to recieve an Alert Report at both work and home, the user account profile may contain one and the other entered here. Alternatively, the user account profile may not have an e-mail address and both are entered here. If entering a process URL, please contact the NCI Help Desk for programming design requirements and instructions.
The following users only receive …
The users who have intentionally requested exclusion from the Recipient Curator groups. These users only receive Alert Report e-mails when included by name or by e-mail address in the Recipient list.
Once a selection is made or e-mail address is entered, select "Add" to insert the address into the recipients. Select a recipient and "Remove" to stop Alert Report e-mail distribution to an address.

Note:During the Automated Generation process, Alert Reports are bundled into a single e-mail per address.
To:
A sample e-mail "To:" address displaying the selected recipients.
Subject:
The "Subject:" appearing on Alert Report distribution e-mails.
(enter static comment here)
Text entered in this field appears in the Report header. It does not appear in the distribution e-mails. As noted above, a recipient may have multiple report references in a single e-mail, consequently the text is included in the report. HTML tags are permitted within the text.
Report Examples
Opens a page containing two (2) Alert Report examples.

Alert Report distribution e-mails contain a brief general introduction and instructions followed by the Alert Report links. Please refer to the Report Output section for more information.

Δ Criteria Tab

The Criteria Tab contains attributes controlling the Alert Report scope. All lists on this page support multiple selections using standard selection key combinations using the mouse, Ctrl and Shift.

Select Context(s)
Limit the report scope to records directly or indirectly associated to one or more Contexts. A record is directly associated when Owned By a Context, e.g. a Data Element. A record is indirectly associated when it is referenced by a record Owned By a Context, e.g. a Value Domain associated to a Data Element in the selected Context.
Select Protocol(s)
Limit the report scope to records directly or indirectly associated to one or more Protocols.
Select Form(s)/Template(s)
Limit the report scope to records directly or indirectly associated to one or more Forms or Templates.
Select Classification Scheme(s)
Limit the report scope to records directly or indirectly associated to one or more Classification Schemes.
Select Classification Scheme Item(s)
Limit the report scope to records directly or indirectly associated to one or more Classification Scheme Items.
Select Administered Component Types
Limit the report scope to specific record types.
Reporting Dates are compared to
Limit the report scope by activity types, created or changed, created only or changed only. This is the only single selection attribute on the page.
Select Workflow Status(es)
Limit the report scope to records with a specific Workflow Status. The Workflow Status must be equal to the selection(s) indicated during report generation to be eligible.
Select Registration Status(es)
Limit the report scope to records with a specific Registration Status. The Registration Status must be equal to the selection(s) indicated during report generation to be eligible.
Select Created By
Limit the report scope to records created by the users selected.
Select Last Modified By
Limit the report scope to records last modified by the users selected. Some records only record the last user applying changes. Others keep a change history with the user making the change. When history records exist the scope includes all changes made by the users selected.

Selections are combined using logical "OR" conditions for an attribute and logical "AND" conditions for the criteria. For example, a selection of Contexts CTEP and CCR and a selection of Created By user Bob Marley will result in a scope set of:

((Context equal "CTEP") or (Context equal "CCR")) and (Created By equal "Bob Marley")

The summary which appears on the Properties tab has been annotated to clearly convey the interpretation of the criteria scope. The parentheses in this example illustrate the logical association of expressions.

Note: The Workflow Status and Registration Status on this tab are not related to the attributes on the Monitors tab. On the Criteria tab these attributes define the current data scope. Selecting a specific value limits the scope using the current value not a previous value. On the Monitor tab these attributes define a specific change to trigger an Alert Report.

Δ Monitors Tab

The Monitors Tab contains attributes controlling the Alert Report triggers. All lists on this page support multiple selections using standard selection key combinations using the mouse, Ctrl and Shift.

Workflow Status(es)
Trigger an Alert Report when the Workflow Status is changed to one of the selections.
Registration Status(es)
Trigger an Alert Report when the Registration Status is changed to one of the selections.
Versoin
Trigger an Alert Report when the Version is changed to one of the selections.

Special selections:

Any Change
Triggers a report for any and all value changes to the record property.
Ignore
Disables triggers for Alert Report creation for the record property.
Whole Number Change
Applies to Version number changes only, triggers a report when the record whole number part changes, decimal number part changes are ignored.
Specific
Applies to Version number changes only, used to indicate a specific version number to trigger the report creation.

Note: The selection combination on the Monitors Tab triggers the Alert Report creation by interrogating the data activities. When all attributes are set to "Any Change", all activity within the scope defined on the Criteria Tab will trigger the report creation. This is the default setting for new Alert Definitions. Selecting specific values for one or more attributes significantly limits that data activities triggering report creation because a specific activity matching the attributes values must be found in the caDSR history for the reporting dates. For example, selecting the Workflow Status "RELEASED" will only trigger a report when this specific change is found in the history - other changes would not trigger a report creation.

Note: Some data is not logged in the caDSR history and will appear as "Details not available" on an Alert Report.

Run Page

The Manual Alert Run page provides input for a report date range and a selection to use either the creator as the sole recipient or the defined recipient list. The request can then be submitted to the server for the single Alert Definition by selecting the Run button from the List and Edit pages.

The process, called a Manual Run, executes on the web server which hosts the Sentinel Tool. A confirmation message appears after submitting the request. Manual Runs always send an e-mail, the "Send a report from …" specified on the Edit page, Report Details tab is ignored. Although the confirmation message appears immediately, the time to create and send the Alert Report depends on many factors including the number of manual runs being processed and the volume of output generated.

A Manual Run executes the same logic and code as an Auto Run with minor exceptions. First, the Manual Run only processes one Alert Definition where the Auto Run may process many. Second, a report is always sent by a Manual Run. When started from the Edit page, an Alert can be run without saving changes to the Definition.

Note: After submitting a Manual Run there is no means to monitor its progress or determine when the Report e-mail will be received.

Dates

Date ranges within the Sentinel Tool and Alerts are always interpreted from midnight to midnight.

  • To manually run an Alert to cover all changes made today, make the starting date the same as today's date and the ending date tomorrow's date.
  • To manually run an Alert to cover all changes made yesterday, make the starting date yesterday's date and the ending date today's date.

Report Output

Report output is controlled by many settings in the Alert Definition.

Reports contain two sections, a header and body. The header always contains the Sentinel Name, Created By Name, Summary, Static Introduction (from the Edit, Report Details tab), Report Date Range and Report Creation Date. Additional information appears as specified by the attributes on the Edit page, Report Details tab. The body contains details relating to the scope and triggers specified by the attributes on the Edit page, Criteria and Monitors tab.

The report body contains several major blocks of information. Each block starts with a Row Number, Group Number, Record Type prefix and Record Name. For example:

1 1 Changes to the Data Element: Blood Pressure

The Row Number shows the number of logical rows of information appearing on the report. This does not represent the number of physical lines. When the Record Type prefix begins with "Changes to the …" the block indicates an actual change to the record being described. When the prefix begins with "Associated to …" the block indicates the record is associated in some way to the most previous change block. These Associated records have not changed. They are provided for the user analysis of possible collateral impacts occurring from the actual change. The Group Number shows the proximity of the change. For example:

1 1 Changes to the Data Element Concept: …
…
2 1.1 Associated To Data Element: …
…
3 1.1.1 Associated To Context: …
…
4 1.2 Associated To Context: …
…
5 2 Changes to the Value Domain: …
…

Rows 1 and 5 are actual changes to records in the caDSR. Note the use of "Changes to the …" and the group number is a simple integer. Rows 2, 3 and 4 are all associated to the change noted in row 1. Row 2 is directly associated to row 1 as noted by the group number 1.1. Row 3 is directly associated to row 2 as noted by the group number 1.1.1 and is indirectly associated to row 1. Row 4 is directly associated to row 1 as noted by the group number 1.2.

Because the main focus of the Alert Report is to document the change activities that occurred in the caDSR, these always precede the associated record blocks. The nature of the associations is such that a true visual representation would require a two-dimensional graphical network diagram. Although the change blocks should only appear once, the associated record blocks most often have duplication.

For each block, change or associated, when possible the report will show public id, version, modified by, modified date, created by and created date. If the public id or version appears as N/A, the record can not have the attribute. If the modified by and date appear as <null>, the record has not been changed since its original creation.

For change activities for most Administered Components (including Data Element, Data Element Concept and Value Domain) another sub-block appears showing the specific changes noted by attribute, old value and new value. Some change blocks state the details are unavailable. This is due to incomplete auditing information and will be addressed with each new release.

Reports are prevented from becoming too large for the browser via a system configuration threshold. When a report exceeds the threshold it is divided into multiple parts. The rows are counted as described above and not as individual lines. This is called a multi-part report. The e-mail sent to each recipient will note the number of parts created. The link provided in the e-mail is to an index of links to each part. Each file in a multi-part report shows the part number with Next and Previous links to facilitate reading and navigating the files. If the report size becomes cumbersome specify a smaller value for the Maximum Associated To Level or make the Alert Definition Criteria more specific.
